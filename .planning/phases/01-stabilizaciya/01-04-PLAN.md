  <action>
Read app.jsx handleCancelRequest (around line 725). The function does:
```js
const { data: updated, error } = await supabase.from('requests').update({ status: 'closed' }).eq('id', reqId).select();
if (error || !updated?.length) { showToast('Ошибка при отмене заявки', 'error'); return; }
```

There are two possible causes for failure under RLS:

1. **The `.select()` after `.update()` returns empty** — With RLS UPDATE USING clause, Supabase filters rows based on the USING condition before applying the update. If the condition passes, the row is updated, but the subsequent `.select()` returns the row only if SELECT policy also allows it. Since SELECT policy is `auth.role() = 'authenticated'`, this should work. However, `.update().eq().select()` can return `{ data: [], error: null }` if the USING condition in UPDATE prevented the update (row not matching).

2. **The `ownerId` column type mismatch** — `auth.uid()::text` returns a string UUID, while `shipperInn` should also be a string UUID (per migration 21). Verify `sbUser.id` is the same format.

Diagnose by adding console.error logging to capture the exact error. Then fix:

**Primary fix:** Remove `.select()` from the update call — it's unnecessary since we do optimistic state update anyway. Change to:
```js
const { error } = await supabase.from('requests').update({ status: 'closed' }).eq('id', reqId);
if (error) {
    console.error('Cancel request error:', error);
    showToast('Ошибка при отмене заявки', 'error');
    return;
}
```

This eliminates the `!updated?.length` false-negative path. The optimistic `setRequests(prev => prev.map(...))` on the next line handles local state.

**Secondary check:** Also verify the client guard at line 728 uses the correct comparison: `req.shipperInn !== sbUser.id`. Both should be UUID strings. If `sbUser` is the Supabase auth user, `sbUser.id` is `auth.uid()` — correct.

Do NOT change RLS policies. Do NOT add new migrations. This is a client-side fix only.
